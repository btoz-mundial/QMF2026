# Archetype Guidelines V2

## Objetivo

Este documento define la filosofía oficial de archetypes del sistema analytics.

Los archetypes existen para:

- resumir patrones de comportamiento
- generar identidad narrativa
- facilitar lectura rápida del usuario
- crear conversación entre participantes
- enriquecer runtime cards y analytics

Los archetypes NO son:

- scoring oficial
- rankings oficiales
- métricas matemáticas absolutas
- diagnósticos definitivos
- clasificación permanente de usuarios

---

# Filosofía General

Los archetypes deben sentirse:

- futboleros
- competitivos
- memorables
- claros
- sociales

NO deben sentirse:

- corporativos
- excesivamente técnicos
- casino-style
- esports extremos
- psicológicos
- exageradamente dramáticos

---

# Principios Fundamentales

## Basados en datos reales

Todo archetype debe derivarse de:

- analytics oficiales
- snapshots oficiales
- runtime insights válidos

NO se permite:

- inventar narrativa
- exagerar desempeño inexistente
- ignorar activation rules
- ignorar sample size

---

## Interpretación flexible

Un archetype representa:

- tendencia general
- estilo competitivo
- patrón observable

NO representa:

- verdad absoluta
- skill definitivo
- categoría fija permanente

Los archetypes pueden cambiar durante el torneo.

---

## Claridad primero

Un usuario debe entender intuitivamente:

- qué comunica el archetype
- por qué lo recibió
- qué comportamiento representa

Si requiere demasiada explicación:
- es demasiado complejo
- demasiado abstracto
- o demasiado técnico

---

# Filosofía de Naming

Los archetypes pueden usar:

- español
- inglés básico
- modismos gaming simples
- referencias deportivas claras

SIEMPRE que:

- sean fáciles de entender
- tengan identidad fuerte
- sean fáciles de recordar
- funcionen socialmente

---

# Tipos de Archetypes

## Archetypes Persistentes

Representan patrones relativamente estables.

Ejemplos:
- Muro de Acero
- Point Hunter
- Table Master

Pueden durar gran parte del torneo.

---

## Archetypes Dinámicos

Representan momentos o tendencias recientes.

Ejemplos:
- En Ascenso
- Caída Fuerte
- Líder Sólido

Pueden cambiar rápidamente entre snapshots.

---

# Relación con Runtime Insights

Los runtime insights:

- detectan eventos
- detectan tendencias
- detectan cambios

Los archetypes:

- resumen identidad competitiva
- sintetizan comportamiento observable

---

# Relación con Tiers

Los archetypes compatibles con tiers pueden utilizar:

| Tier | Formato |
|---|---|
| ⭐ | Archetype |
| ⭐⭐ | Archetype Elite |
| ⭐⭐⭐ | Archetype Supremo |

---

# Filosofía de Escasez

Los archetypes más fuertes deben mantenerse relativamente escasos.

Si demasiados usuarios reciben:
- Elite
- Supremo
- dominance archetypes

el sistema pierde:
- identidad
- impacto
- competitividad

---

# Compatibilidad con Activation Rules

Un archetype NO debe asignarse si:

- el analytic base aún no es stable
- el sample size es insuficiente
- existe alta volatilidad estadística
- los datos aún no son interpretables

---

# Archetypes y Timeline Race

Timeline Race es uno de los principales motores narrativos del sistema.

Puede derivar:

- momentum
- remontadas
- estabilidad
- liderazgo
- supervivencia competitiva

Esto permite generar archetypes dinámicos sin alterar scoring oficial.

# Objetivo Final

Los archetypes deben ayudar a que la quiniela se sienta como:

“una competencia futbolera inteligente entre amigos donde cada usuario desarrolla una identidad competitiva basada en analytics reales y auditables.”
