# Analytics Tiering Guidelines V2


## Objetivo

Este documento define la filosofía oficial de tiers para analytics, runtime cards, rankings y archetypes.

Su propósito es:

- mantener consistencia narrativa
- evitar inflación de tiers
- permitir flexibilidad durante el prototipo 2026
- proteger claridad para usuarios
- mantener engagement saludable
- evitar sistemas rígidos prematuros

Este documento NO define:

- thresholds exactos permanentes
- percentiles fijos globales
- fórmulas hardcodeadas
- reglas matemáticas definitivas

---

# Filosofía General

Los tiers existen para:

- destacar desempeño
- generar identidad
- incentivar competencia sana
- mejorar narrativa social
- hacer más memorables los analytics

Los tiers NO deben:

- exagerar resultados
- inflar desempeño promedio
- parecer sistema casino
- sentirse pay-to-win
- convertirse en ruido visual

---

# Estructura Oficial de Tiers

Todos los analytics compatibles con tiers utilizan la siguiente estructura:

| Nivel | Formato |
|---|---|
| ⭐ | Gaming Name |
| ⭐⭐ | Gaming Name Elite |
| ⭐⭐⭐ | Gaming Name Supremo |

---

# Filosofía de cada tier

## ⭐ Tier Base

Representa:

- desempeño destacado válido
- resultados positivos consistentes
- métricas competitivas

Debe sentirse:

- alcanzable
- satisfactorio
- competitivo

NO debe sentirse:
- automático
- regalado
- genérico

---

## ⭐⭐ Elite

Representa:

- desempeño claramente superior
- consistencia notable
- diferenciación competitiva real

Debe sentirse:

- difícil de conseguir
- respetado
- competitivo

Elite NO debe inflarse excesivamente.

---

## ⭐⭐⭐ Supremo

Representa:

- desempeño excepcional
- resultados poco comunes
- dominio claro dentro del analytic

Debe sentirse:

- raro
- especial
- memorable

Supremo debe mantenerse relativamente escaso para preservar valor narrativo.

---

# Flexibilidad del prototipo 2026

Durante la edición prototipo 2026:

- los thresholds pueden ajustarse dinámicamente
- los percentiles pueden evolucionar
- el frontend puede adaptar distribución de tiers
- Claude puede ayudar a interpretar desempeño relativo

Esto permite:

- observar comportamiento real
- detectar inflación
- validar engagement
- calibrar dificultad real

---

# Restricciones importantes

Aunque exista flexibilidad:

## NO se permite:

- asignar tiers aleatoriamente
- inventar desempeño inexistente
- ignorar analytics oficiales
- exagerar métricas débiles
- regalar tiers masivamente

---

# Filosofía de asignación

La asignación debe considerar:

- percentiles relativos
- distribución real de usuarios
- sample size válido
- estabilidad estadística
- contexto del torneo

---

# Escasez narrativa

Los tiers más altos deben sentirse especiales.

Un sistema donde:

- todos son Elite
- muchos son Supremo

pierde:
- impacto
- emoción
- competitividad
- identidad

---

# Compatibilidad con analytics

No todos los analytics requieren tiers.

Los tiers deben aplicarse preferentemente a:

- precision_general
- precision_avance
- precision_marcadores_exactos
- eficiencia_de_puntos
- consistencia_ranking
- timeline_race

---

# Analytics con precaución

Analytics altamente volátiles pueden requerir restricciones:

Ejemplos:
- métricas con sample pequeño
- eliminatorias tempranas
- rankings inestables
- trends recientes

---

# Relación con activation rules

Un analytic NO debe asignar tiers si:

- aún no alcanza estado stable
- tiene confidence bajo
- el sample es insuficiente
- existe alta volatilidad estadística

---

# Filosofía visual implícita

Aunque este documento NO controla UI:

Los tiers deben sentirse:

- deportivos
- modernos
- limpios
- competitivos

NO:
- tragamonedas
- casino
- MMORPG exagerado
- esports extremo

---

# Objetivo final

Los tiers deben reforzar la sensación de:

“competencia futbolera inteligente entre amigos”

sin perder:

- transparencia
- claridad
- credibilidad
- auditabilidad
- identidad narrativa.
