# QM2026 Experience Framework

## Purpose

QM2026 is not an analytics dashboard.

It is a competitive tournament experience built around:
- identity
- prediction survival
- leaderboard pressure
- tournament storytelling
- transparency
- competitive evolution

The system must feel like:
- a premium fantasy football experience
- a live tournament broadcast
- a competitive ecosystem

NOT:
- a SaaS analytics panel
- a spreadsheet viewer
- a metrics-heavy admin interface



# Core Product Philosophy

The platform is structured around four distinct modes of interaction with the tournament:

| Section | Purpose | Emotional Question |
|---|---|---|
| Tabla | Official tournament leaderboard | “¿Cómo va el leaderboard?” |
| Player | Personal competitive experience | “¿Cómo va mi torneo?” |
| Resultados Oficiales | Official World Cup coverage | “¿Qué ha pasado realmente?” |
| Analytics | Competitive ecosystem exploration | “¿Por qué vamos como vamos?” |

Each section must feel emotionally distinct.

The user should always understand:
- what world they are in
- what perspective they are seeing
- what emotional role the section serves



# Navigation Philosophy

Navigation is NOT organized around datasets.

Navigation is organized around:
- emotional context
- competitive perspective
- tournament relationship

The experience separates:
- official tournament state
- personal tournament survival
- competitive interpretation

This separation is critical.



# Section Definitions



## TABLA

### Role
Official competitive leaderboard.

### Emotional Purpose
“¿Cómo va el leaderboard?”

### Philosophy
Tabla represents:
- the official state of the competition
- rank pressure
- payout race
- tournament positioning

It is the broadcast-facing center of the tournament.

### Must Feel Like
- live standings
- premium competition
- official tournament coverage

### Must NOT Feel Like
- analytics dashboard
- player profile page
- prediction explorer



## PLAYER

### Role
Personal tournament experience.

### Emotional Purpose
“¿Cómo va mi torneo?”

### Philosophy
Player is the emotional heart of the product.

This is where the user experiences:
- their competitive identity
- their rise/fall
- their prediction survival
- their payout race
- their tournament story

Player is NOT:
- an account page
- a settings page
- a gamer profile
- a stats dump

Player is:
- a live competitive narrative

The profile should answer:

> “What kind of tournament is this player having?”

before:
- metrics
- detailed scoring
- raw analytics

### Core Sections
- Resumen
- Predicciones
- Timeline
- Auditoría

#### Resumen Philosophy
Phase Performance should become the dominant competitive visualization inside Resumen.

The purpose is to preserve competitive identity and engagement beyond global ranking position.

Even players outside payout contention should still feel:
- competitive relevance
- specialization
- tournament strengths
- surviving identity

Resumen should avoid reducing the tournament experience to:
- only global rank
- only payout pressure
- only leaderboard survival

The product should allow users to feel:
> “I may not be leading the tournament, but I still have a meaningful tournament identity.”

### Subtab Emotional Questions

| Subtab | Emotional Question |
|---|---|
| Resumen | “¿Qué tal me va en el torneo?” |
| Predicciones | “¿Qué tal van mis apuestas?” |
| Timeline | “¿Cómo va mi carrera por el premio?” |
| Auditoría | “¿Puedo confiar en este torneo?” |




# HeroIdentity Philosophy

The HeroIdentitySection is the dominant emotional anchor of the Player experience.

The archetype layer must NOT behave like metadata.

It must behave like:
- competitive identity
- tournament personality
- emotional framing

The HeroIdentitySection communicates simultaneously:
- who the player is
- what type of tournament they are having

The archetype should visually coexist with:
- rank
- points
- tournament position

Traits remain subordinate.

identity_formula carries more emotional weight than:
- short_description
- supporting metrics

The visual hierarchy must prioritize:
- identity
- atmosphere
- competitive tension
- breathing room



## RESULTADOS OFICIALES

### Role
Official tournament coverage.

### Emotional Purpose
“¿Qué ha pasado realmente?”

### Philosophy
Resultados Oficiales represents:
- the official World Cup progression
- the UI display of the source of truth
- real tournament outcomes

This section contains subtabs:
- Grupos
- Bracket

Both operate as:
- official tournament views
- not personal dashboards

### Important Overlay Philosophy

Official tournament views may include:
- lightweight player overlays
- prediction survival overlays
- point indicators

BUT:
the official tournament structure must remain dominant.

The player layer is secondary to the tournament layer.

### Must Feel Like
- premium sports coverage
- official FIFA-style progression
- tournament reality

### Must NOT Feel Like
- a personal dashboard
- prediction overload
- score annotation spam



## ANALYTICS

### Role
Competitive ecosystem exploration.

### Emotional Purpose
“¿Por qué vamos como vamos?”

### Philosophy
Analytics is NOT:
- a metrics dump
- a developer analytics page
- a spreadsheet explorer

Analytics is:
- interpretation
- ecosystem exploration
- competitive storytelling

This section exists to:
- discover archetypes
- compare competitive styles
- identify specialists
- explore volatility
- understand tournament evolution
- explore tournament behavior patterns

Analytics should feel like:
- exploring the league
- understanding the ecosystem
- studying tournament dynamics

NOT:
- browsing charts endlessly



# Visual Direction Philosophy

The experience should feel:
- premium
- atmospheric
- cinematic
- competitive
- broadcast-inspired

NOT:
- corporate
- admin-like
- SaaS-oriented
- spreadsheet-heavy

The interface should prioritize:
- hierarchy
- spacing
- pacing
- emotional composition
- tournament tension



# Layout Philosophy

The product should avoid:
- compressed vertical layouts
- equal-weight cards
- metric overload
- dense dashboard stacking

The experience should breathe.

Sections should feel:
- editorial
- cinematic
- deliberate

NOT:
- compressed
- technical
- crowded



# Runtime Philosophy

The frontend is render-only.

The frontend must NOT:
- derive competitive identity
- calculate hidden logic
- infer narrative systems
- fabricate momentum
- create editorial interpretation

All competitive interpretation must originate from:
- deterministic analytics
- auditable runtime outputs
- official tournament data



# Transparency Philosophy

Transparency is a core product pillar.

Users must be able to:
- audit scoring
- understand rankings
- verify prediction freezes
- inspect tournament calculations
- trust leaderboard legitimacy

Transparency is NOT secondary UX.

It is a competitive feature.



# Anti-Goals

QM2026 must NOT become:
- an admin dashboard
- a generic sports stats app
- a casino UI
- a gamified RPG interface
- an achievement spam system
- a metrics overload platform
- a spreadsheet disguised as a product

The experience must preserve:
- clarity
- emotional hierarchy
- tournament atmosphere
- competitive seriousness



# Narrative Hierarchy

The system should always prioritize:
1. Tournament identity
2. Competitive tension
3. Personal survival
4. Leaderboard pressure
5. Ecosystem interpretation
6. Detailed analytics

NOT the reverse.



# Future Expansion Rules

New systems should only exist if they strengthen:
- tournament storytelling
- competitive clarity
- emotional pacing
- ecosystem depth

Avoid:
- feature accumulation
- unnecessary tabs
- analytics duplication
- UI fragmentation

Every new feature must answer:
> “What emotional role does this serve in the tournament experience?”



# Current Product Bottleneck

The current bottleneck of the project is NO LONGER:
- analytics
- contracts
- scoring systems

The current bottleneck is:
- visual storytelling
- emotional hierarchy
- cinematic composition
- pacing
- tournament atmosphere

Future frontend work should prioritize:
- composition
- emotional clarity
- hierarchy refinement
- narrative presentation

before adding:
- more analytics
- more runtime systems
- more UI density
- more tabs
